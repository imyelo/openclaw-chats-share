# Chats Share Data Format

This directory contains Openclaw session export files in YAML format, generated by the CLI from `session.log` JSONL files.

## File Naming

Use the format: `YYYYMMDD-{slug}.yaml`

Example: `20260215-debugging-async-issue.yaml`

## Top-level Metadata Fields

| Field | Source | Required | Description | Example |
|-------|--------|----------|-------------|---------|
| `title` | `FormatConstraint.name` | Yes | Session title / export name | `My Session` |
| `date` | `session.timestamp` | Yes | Session date (YYYY-MM-DD) | `2026-02-15` |
| `sessionId` | `session.id` | Yes | Unique session ID | `cf1f8dbe-2a12-47cf-8221-9fcbf0c47466` |
| `channel` | Manual | No | Channel/platform name | `discord`, `telegram` |
| `model` | First message's `model` field | No | Model used in session | `MiniMax-M2.5` |
| `totalMessages` | Count of user+assistant messages | No | Excludes toolResult | `42` |
| `totalTokens` | Sum of `usage.totalTokens` | No | Total tokens consumed | `12345` |
| `tags` | Manual | No | Tag array for categorization | `[coding, debug]` |
| `visibility` | Manual | No | Controls index visibility (`public` or `private`) | `private` (default) |
| `defaultShowProcess` | CLI / Manual | No | If `true`, process blocks are shown by default | `false` |
| `description` | Manual | No | Brief description for index display | `Debugging a tricky async issue` |
| `participants` | CLI / Manual | No | Maps each participant name to `{ role: "human" \| "agent" }` | see example below |

## Visibility

- `visibility: public` — Chat will appear on the homepage index
- `visibility: private` (default) — Chat will NOT appear on the homepage index, but remains accessible via direct URL

## Timeline Array

After the metadata fields, a `timeline:` key holds an ordered list of event and message objects sorted by `timestamp`.

### Non-message Event Types

| `type` | Additional fields | Description |
|--------|-------------------|-------------|
| `session` | `cwd` | Session start event |
| `model_change` | `model`, `provider` | Model switch event |
| `thinking_level_change` | `level` | Thinking level change (e.g. `"off"`, `"medium"`) |
| `compaction` | `tokensBefore` | Context compaction event |
| `custom` | `customType`; for `model-snapshot`: `model`, `provider` | Custom events |

All events also include `timestamp`.

**Optional fields available on all event types:**

| Field | Description |
|-------|-------------|
| `title` | Human-readable label shown as the collapsible panel header. Overrides the auto-generated label. Suitable for manual curation or AI summarization. |
| `speaker` | Name of the participant who triggered the event (matched against `participants` map). Enables the event to be grouped under that participant's header. |
| `content` | Markdown body shown when the panel is expanded. |

### Message Objects

Each message entry may carry a `process` list (thinking blocks, tool calls) and/or a `content` field. The CLI emits each process block and the text content as **separate entries** with the same `speaker`, `timestamp`, `role`, and `model`. Hand-crafted YAML may consolidate multiple blocks into a single entry — the web handles both forms.

**Entry with a thinking block:**

```yaml
- type: message
  role: agent
  speaker: {DisplayName}
  timestamp: {ISO 8601}
  model: {model-name}
  process:
    - type: thinking
      content: |
        {reasoning text}
      collapsed: false          # optional; if true, the panel starts collapsed
```

**Entry with a tool call:**

```yaml
- type: message
  role: agent
  speaker: {DisplayName}
  timestamp: {ISO 8601}
  model: {model-name}
  process:
    - type: tool_call
      id: {tool-call-id}
      name: {tool-name}
      title: {display label}    # optional; overrides the auto-generated "Tool Call - name · path" label
      collapsed: false          # optional
      arguments:
        {key}: {value}
      result:
        content: |
          {result text}
        isError: false
```

**Entry with text content:**

```yaml
- type: message
  role: human | assistant
  speaker: {DisplayName}
  timestamp: {ISO 8601}
  model: {model-name}           # assistant only
  content: |
    {message text}
  images:
    - mimeType: image/png
      data: {base64}
```

Block scalars (`|`) are used for multiline string fields (`content`, result `content`). Single-line strings use plain or quoted YAML style.

## Complete Example

```yaml
title: Debugging Async Issue
date: 2026-02-15
sessionId: cf1f8dbe-2a12-47cf-8221-9fcbf0c47466
model: MiniMax-M2.5
totalMessages: 4
totalTokens: 12345
visibility: public
defaultShowProcess: false
participants:
  Alice:
    role: human
  Claude:
    role: agent
    model: claude-sonnet-4-6

timeline:
  - type: session
    timestamp: "2026-02-15T06:13:00.000Z"
    cwd: ~/projects/myapp

  - type: model_change
    timestamp: "2026-02-15T06:13:01.000Z"
    model: MiniMax-M2.5
    provider: minimax

  - type: message
    role: human
    speaker: Alice
    timestamp: "2026-02-15T06:13:50.514Z"
    content: |
      Message content...

  # CLI emits one entry per block; hand-crafted YAML may combine them.

  - type: message
    role: agent
    speaker: Claude
    timestamp: "2026-02-15T06:14:05.123Z"
    model: claude-sonnet-4-6
    process:
      - type: thinking
        content: |
          Let me reason through this...

  - type: message
    role: agent
    speaker: Claude
    timestamp: "2026-02-15T06:14:05.123Z"
    model: claude-sonnet-4-6
    process:
      - type: tool_call
        id: tc_001
        name: read
        arguments:
          file_path: src/emitter.ts
        result:
          content: |
            // file contents
          isError: false

  - type: message
    role: agent
    speaker: Claude
    timestamp: "2026-02-15T06:14:05.123Z"
    model: claude-sonnet-4-6
    content: |
      Response content...

  - type: custom
    timestamp: "2026-02-15T06:14:10.000Z"
    customType: model-snapshot
    model: MiniMax-M2.5
    provider: minimax

  - type: message
    role: human
    speaker: Alice
    timestamp: "2026-02-15T06:15:00.000Z"
    content: |
      Can errors be shown?

  - type: message
    role: agent
    speaker: Claude
    timestamp: "2026-02-15T06:15:10.000Z"
    model: claude-sonnet-4-6
    process:
      - type: tool_call
        id: tc_002
        name: read
        arguments:
          file_path: /nonexistent/file.txt
        result:
          content: File not found
          isError: true

  - type: message
    role: agent
    speaker: Claude
    timestamp: "2026-02-15T06:15:10.000Z"
    model: claude-sonnet-4-6
    content: |
      Of course, errors are displayed in a red panel.
```

## Web UI Rendering

The web package reads `.yaml` files from the `chats/` directory (or `chats_dir` from `chats-share.toml`) and renders them as a conversation view. Each timeline item is rendered as follows:

| Item | Rendering |
|------|-----------|
| `message` with `content` | Chat bubble |
| `message` with `process[].type = thinking` | Collapsible gray panel (expanded by default; set `collapsed: true` to collapse) |
| `message` with `process[].type = tool_call` | Collapsible indigo panel per call (red panel if `result.isError: true`) |
| `session` event | Collapsible green panel |
| `model_change` event | Collapsible indigo panel |
| `thinking_level_change` event | Collapsible gray panel |
| `compaction` event | Collapsible indigo panel |
| `custom` event | Collapsible indigo panel |

The "show process" toggle button hides or reveals all collapsible panels globally.
